# EAIRA Standard Project Task Specification

## Title

EAIRA Standard Project Task Specification

## Purpose

Define the shared structure, section purposes, and writing principles for Project Layer task documents.

This specification normalizes how project tasks are written. It does not define governance policy, create execution records, or authorize work.

## Specification Scope

This specification applies to task documents within the Project Layer only.

It defines:

- Shared task structure.
- Section purposes.
- Writing principles.
- Compliance expectations.
- Evolution guidance for the project task format.

It does not apply to Governance Layer documents and does not modify governance documents.

## Standard Task Structure

Project Layer task documents should use this structure unless a documented reason to deviate exists:

1. Task.
2. Objective.
3. Inputs.
4. Scope.
5. Rules.
6. Non-Goals.
7. Domain Specific Sections.
8. Required Content.
9. Acceptance.
10. Final Decision.

## Section Purposes

| Section | Purpose |
| --- | --- |
| Task | State what the document is for. |
| Objective | Define the outcome to be achieved. |
| Inputs | List the evidence or source documents. |
| Scope | Define what will be created or updated. |
| Rules | Define execution constraints. |
| Non-Goals | Define explicit exclusions. |
| Domain Specific Sections | Define task-specific content that does not belong in the standard sections. |
| Required Content | Define required outputs in the document. |
| Acceptance | Define completion criteria. |
| Final Decision | Define the standardized decision format. |

## Writing Principles

- Keep the structure consistent across all Project Layer task documents.
- Use the standard task structure unless a task has a documented reason to deviate.
- Keep domain-specific content in the middle sections only.
- Preserve evidence-first and historical-record behavior where applicable.
- Maintain separation between Project Layer and Governance Layer.
- State constraints directly and avoid implied authorization.
- Record only supported project facts when documenting history.

## Compliance

A Project Layer task document is compliant when:

- It follows the standard task structure.
- It clearly defines scope and non-goals.
- It uses domain-specific sections only where appropriate.
- It maintains separation between Project Layer and Governance Layer.
- It identifies inputs for claims that depend on existing project history.
- It avoids creating records or decisions outside the task scope.

## Specification Evolution

This specification may evolve as the Project Layer matures.

Changes should improve consistency, maintainability, and reuse across project task documents. Changes should be based on validated project experience rather than speculative design.

## Example Task Types

The following are representative examples rather than a complete list:

- Project Charter.
- Project Progress Report.
- Project Status Report.
- Roadmap.

## Final Decision

EAIRA Standard Project Task Specification is documented as a project-layer master specification.
